# Scroll Peddler 에이전트 작업 지침

이 파일은 저장소 전체에 적용한다. 에이전트와 자동화 도구는 작업을 시작하기 전에 이 문서와 변경 대상 경로의 추가 지침을 확인한다.

## 소통과 명명

- 사용자 보고, 프로젝트 문서, 커밋 설명은 한국어를 기본으로 한다.
- 기존 `Docs/TECH_SPIKE.md`는 이번 기록 정리에서 현재 언어를 유지하며, 별도 요청 없이 번역만을 위한 변경을 하지 않는다.
- 코드 식별자, 파일명, 로그 태그, 콘솔 명령, 브랜치명은 영어를 유지한다.
- 커밋 제목은 영문 Conventional Commit 타입과 한국어 설명을 조합한다. 예: `feat: 서버 권위형 상호작용 추가`.

## 확정 기술과 범위

- Unreal Engine 5.8 계열의 Unreal C++ 프로젝트이며 JetBrains Rider를 기본 IDE로 사용한다.
- Windows PC와 Steam 배포를 우선한다.
- 네트워크 모델은 1–4인 Listen Server와 호스트 권위형 협동이다.
- 소스와 문서는 GitHub, Unreal 바이너리 에셋은 Git LFS로 관리한다.
- MVP에서는 자체 백엔드, 전용 서버, Iris, Online Services를 사용하지 않는다.
- 현재 기술 스파이크는 Steam 세션이 아닌 raw IP Listen Server 접속이다. Steam 로비·세션·친구 초대가 구현된 것으로 간주하지 않는다.

## 구현 책임 경계

- 게임 규칙, 권위 판정, 복제되는 상태, 저장 및 정산 무결성은 C++가 책임진다.
- 수치, 스크롤 정의, 각인, 드롭 구성처럼 반복 조정할 콘텐츠는 Data Asset 또는 Primary Data Asset으로 둔다.
- 화면 표현, 연출, VFX, UI와 제작자가 배치할 월드 콘텐츠는 Blueprint와 Unreal Editor 에셋을 적극 활용한다.
- 현재 코드로 생성하는 회색 상자 오브젝트는 기술 스파이크용 임시 구성이다. 최종 콘텐츠 제작 방식으로 확장하지 않는다.
- 메시·머티리얼·에셋 이름에 게임 규칙을 결합하지 않는다. 임시 아트는 교체 가능한 데이터와 soft reference를 우선한다.

## 서버 권위와 네트워크

- 클라이언트는 줍기, 버리기, 사용, 제작, 계약, 탈출 같은 의도만 전송한다.
- 서버는 결과를 확정하고 복제한다. 클라이언트 예측은 되돌릴 수 있는 표현과 로컬 피드백으로 제한한다.
- 서버 RPC는 최소한 거리, line of sight, 소유권, 인벤토리 용량, 플레이어·세션 상태, 요청 중복을 검증한다.
- 동시에 같은 아이템을 요청하거나 RPC를 재전송해도 중복 획득·중복 소비·중복 정산이 발생하지 않아야 한다.
- 완료로 확정된 저장과 정산은 재시도 가능한 멱등 흐름으로 설계하며, ACK 재전송 때문에 중복 반영되거나 영구 손실되지 않아야 한다.
- 현재 스파이크는 추출 또는 save-ACK 중 연결이 끊기면 run을 미완료로 남기며 재접속·재개를 지원하지 않는다. 이 한계를 해결하기 전에는 연결 복구가 구현됐다고 보고하지 않는다.
- 거부 결과는 원인을 확인할 수 있는 로그 또는 사용자 피드백을 남긴다. 권위 판정을 클라이언트 전용 코드에 숨기지 않는다.

## 스크롤 데이터 모델

- 기본 계열, 속성 각인, 품질 D–S, 오염, 오작동을 서로 독립적인 축으로 유지한다.
- 각인을 품질·오염·오작동의 대체 표현으로 사용하지 않는다.
- 조합별 클래스를 복제하기보다 데이터와 조합 가능한 효과 단위로 확장한다.
- 저장 데이터가 참조하는 Stable ID의 의미를 보존한다. Data Asset 이름이나 경로를 바꿀 때 기존 SaveGame이 조용히 무효화되지 않는지 확인한다.

## 에셋과 생성 파일

- `.uasset`과 `.umap`은 텍스트나 일반 바이너리 편집기로 수정하지 않는다. Unreal Editor 또는 검토 가능한 Editor Python 작업으로 변경한다.
- `.gitattributes`가 지정한 바이너리 형식은 Git LFS 포인터로 추적한다. 커밋 전 `git lfs status`로 확인한다.
- `Binaries`, `Intermediate`, `Saved`, `DerivedDataCache`, IDE 상태와 패키징 결과물은 커밋하지 않는다.
- 에디터 자동 생성물을 소스 에셋으로 오인하지 않는다. 재현 가능한 생성 스크립트는 `Scripts/`에 두고 입력과 결과를 문서화한다.

## 변경 안전 수칙

- 작업 전 `git status`와 관련 diff를 확인하고, 사용자 또는 다른 작업자의 관련 없는 변경을 보존한다.
- 명시적 승인 없이 기존 파일·에셋을 삭제하거나 이동하지 않는다. 되돌리기 위해 `git reset --hard`나 무차별 checkout을 사용하지 않는다.
- 토큰, 비밀번호, 인증서, 개인 경로 같은 비밀정보를 커밋하지 않는다.
- Android File Server는 비활성 상태를 유지한다. `ScrollPeddler.uproject`에서 플러그인을 활성화하지 않고, `DefaultEngine.ini`의 `bEnablePlugin=False`, `bAllowNetworkConnection=False`, 빈 `SecurityToken=`을 유지한다.
- 자체 백엔드, 전용 서버, Iris, Online Services 또는 프로덕션 의존성·플러그인을 추가하려면 먼저 이유와 영향을 문서화하고 사용자 승인을 받는다.

## 현재 구현 우선순위

다음 순서를 `README.md`의 구현 순서보다 상세한 기술 게이트로 삼고, 동시에 진행 중인 핵심 P0/P1 작업은 하나로 제한한다.

1. 최종 1인칭 카메라·손·원격 플레이어 바디 구조
2. 상호작용의 로컬 피드백과 서버 수락·거부 경계 분리
3. 거리·LOS·소유권 위조·동시 claim을 포함한 적대적 멀티플레이 테스트
4. 패키지 호스트·클라이언트 다중 프로세스 smoke 안정화
5. G1/G2 성공 기준 문서화와 소리에 반응하는 회색 상자 위협을 포함한 핵심 루프 플레이테스트

새 콘텐츠 수를 늘리기 전에 위 순서의 권위 경계와 검증 게이트를 통과한다.

## 검증과 완료 기준

- C++ 또는 빌드 설정을 바꾸면 Win64 Development Editor 빌드를 통과시킨다.
- 게임 규칙, 데이터 검증 또는 저장 코드를 바꾸면 `ScrollPeddler` Unreal 자동화 테스트를 실행한다.
- 네트워크, 세션, 저장, 정산 또는 핵심 게임플레이를 바꾸면 패키지 호스트·클라이언트 2프로세스 smoke를 실행한다. 4인 경로가 구현되면 4프로세스 검증도 추가한다.
- 바이너리 에셋을 바꾸면 `git lfs status`와 LFS 포인터 추적 여부를 확인한다.
- 문서만 바꾼 경우 `git diff --check`, 링크와 명령 예시, 비밀정보 노출 여부를 확인한다.
- 기능 완료에는 정상 경로뿐 아니라 거부·오류 피드백, 저장 영향, 관련 테스트가 포함된다.
- 완료 보고에는 실제 실행한 검증과 결과, 실행하지 못한 검증, 남은 한계를 구분해 기록한다.
- 현재 빌드·자동화·패키징 명령은 `README.md`와 `Docs/TECH_SPIKE.md`를 기준으로 한다.

## Git 작업 규칙

- 이 초기 기록 정리 이후에는 `agent/<english-kebab-case>` 형식의 작업 브랜치를 사용한다.
- 기능, 테스트, 설정, 문서를 독립적으로 되돌릴 수 있는 논리 커밋으로 나눈다.
- 혼합된 작업 트리에서는 파일을 명시해 stage하고 `git add -A`를 사용하지 않는다.
- 커밋 타입은 `feat`, `fix`, `refactor`, `test`, `docs`, `chore`, `build`, `ci`를 기본으로 한다.
- 커밋 제목은 `fix: 정산 중복 반영 방지`처럼 영문 타입 뒤에 한국어 설명을 쓴다.
- `.uasset`과 `.umap`은 의존하는 기능과 함께 커밋하여 중간 커밋이 깨진 상태가 되지 않게 한다.
- 이번 초기 정리 이후 `main`에는 강제 푸시하지 않는다.
- 변경 보고와 PR 설명은 한국어로 작성하고 변경 내용, 검증 결과, 알려진 한계를 포함한다.
